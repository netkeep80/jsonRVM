38 Формат json не является языком программирования МО, поэтому для исполнения
39 МО требуется десериализация json в сегмент данных РВМ, а так же загрузка базовых
40 словарей сущностей для данного типа РВМ.

43 #/RVM/exec
44 Вычисление результата исполнения экземпляра отношения происходит через интерпретацию
45 сущности (либо исполнение её json проекции) выступающей в качестве сущности-отношения
46 в данном экземпляре отношения. Что бы исполнить сущность, у неё должна быть проекция в
47 язык который понимает вычислитель (РВМ), например:
48 если используется JavaScript виртуальная машина, то у сущности должна быть
49 проекция в JavaScript. Если у сущности нет проекции в язык понимаемый имеющимся
50 вычислителем, то необходимо либо скомпилировать тело сущности (т.е. создать
51 проекцию в нужный язык), либо интерпретировать экземпляр отношения которым она является.
52 Сущность при исполнении в качестве отношения в экземпляре отношения может
53 создать в контексте свойство с именем языка исполнителя и скомпилировать себя
54 туда для непосредственного исполнения сейчас и/или в следующий раз.
55 
56 При исполнении МО существует два потока:
57  иерархически вертикальный - это поток байткода json
58  иерархически горизонтальный - это поток данных в json.
59 Горизонтальные потоки это межслойное преобразование данных контроллерами MVC.
60 Вертикальные потоки это потоки управления интерпретацией json-байткода
61 контроллеров MVC виртуальной машиной.

64 #/Entity/ExecContext
65 Контекст исполнения сущности необходим для её исполнения, т.е. исполнения экземпляра отношения.
66 При исполнении МО контексты образуют собой древовидный стэк. Это достигается за счет того, что
67 каждый контект имеет ссылку на родительский контекст.
68 
69  Строковая адресация в будущей версии jsonRVM
70 
71 Строка содержит в себе последовательную запись инструкций по перемещению указателя сущности в МО.
72 По умолчанию указатель установлен на текущую проекцию сущности.
73 Путь сущности — это набор действий пути. Действия пути могут содержать следующие элементы и операторы.
74 
75 Знак доллара ($) указывает на смену контекстного адресного пространства.
76 После него должно следовать имя контекста.
77 
78 Имена ключей. Например, $.name или $."first name". Если имя ключа начинается со знака доллара или
79 содержит специальные символы, например пробелы, заключите его в кавычки.
80 Элементы массива. Например, $.product[3]. Массивы отсчитываются от нуля.
81 Оператор "точка" (.) указывает на свойство сущности.
82 Например, в $.people[1].surname surname является дочерним элементом people.
83 
84 Знак октоторп (#) указывает на текущая модель отношений в глобальном пространство имён, примеры: "#.global_var"
85 
86 "$ent" - контекстная сущность, примеры: "$ent.$obj", "$ent.some_flag"
87 "$sub" - контекстный субъект , примеры: "$sub.some_attribue", "$sub[43]"
88 "$obj" - контекстный объект, примеры: "$obj.arg1", "$obj[1]"
89 "", "$view" - (по умолчанию) локальное адресное пространство , т.е. текущее значение проекции сущности, примеры: ".local_var", "$view.[353]"
90 
91 "$$ent" - родительская контекстная сущность, примеры: "$$ent.$obj", "$$ent.some_flag"
92 "$$sub" - родительская контекстный субъект , примеры: "$$sub.some_attribue", "$$sub[43]"
93 "$$obj" - родительская контекстный объект, примеры: "$$obj.arg1", "$$obj[1]"
94 "$$view" - родительское локальное адресное пространство, т.е. текущее значение проекции сущности, примеры: "$$view.local_var", "$$view.[353]"

101 #/RM/jsonView/types
102  Соответсвие json значений и проекций Модели Отношений:
103 
104 1. Number - стабильная топологически замкнутая (т.е. сама в себя) проекция отношения double|int32|int64|uint32|uint64
105 2. Boolean - стабильная топологически замкнутая проекция отношения bool
106 3. String - стабильная топологически замкнутая проекция отношения string, либо имя сущности
107 4. Object - стабильная топологически замкнутая проекция отношения вмещения структуры, либо сущность если есть поля "$obj" или "$rel" или "$sub" или "/"
108 5. Array - стабильная топологически замкнутая проекция цепочки отношений последовательного следования - ","
109 6. Null - пустое пространство как потенциал вмещения проекции
110 
111 Null и Array - это не совсем типы, они принципиально отличаются от типов и их обработка идёт отдельными кейсами if в РВМ:
112 1. Array - это контейнер проекцией одного или разного типа
113 2. Null - это отсутствие проекции любого типа
114 
115 Начало цепочки отношений "," всегда есть Null:
116 ((Null --","-> SomeEnt1) --","-> SomeEnt2) === jsonArray[SomeEnt1, SomeEnt2]
117 Это объясняется тем, что начало цепочки это то место перед которым нет другого
118 начального элемента, т.е. пустое пространство обозначаемое через Null.

122  Алгоритм исполнения json после загрузки:
123 
124 1. json после десериализации непостредственно представляет собой сегментом данных RVM
125 2. корневое значение json всегда есть корневая сущность загруженной Модели Отношений,

127 3. поля указывающие на субъект, отношение и объект могут либо содержать
128  сущность по значению (типы json значений: объект, массив)
129  либо содержать ссылки на сущность (типы json значений: строка, число)
130  либо признак ссылки на контекстную сущность (типы json значений: null)
131  либо значение boolean
132 4. Местоимения при нахождении их проекции возвращают ссылку на элементы контекста,
133  проекции которых были найдены до вызова контроллера.

136 #/exec
137 Самое основное отношение это отношение "#/exec", оно исполняет сущность объекта указанного
138 в поле "$obj" и записывает полученную проекцию в поле "$sub" субъекта.
139 Именно это отношение исполняется для кажого поля json объекта, когда он указан
140 в качестве отношения сущности.

142 ------------------------------------------------------------------------------
143  Правила json скрипта (проекции МО в json):
144 
145 1. json представляет собой агрегат проекций сущностей т.е. иерархию МО по сущности-субъекту,
146 2. местоимение субъекта "$sub" может являеться сущностью агрегатом проекции
147 3. запуск скрипта это вычисление проекции корневой сущности
148 4. внутри корневой json-сущности лежат разные проекции её свойств, либо...
149 5. внутри корневой json проекции лежит jsonView корневой сущности
150 6. у локальных не корневых проекций субъект "$sub" равен сущности в которой они лежат
151 7. если в json объекте есть поле "$rel", то значит этот объект описывает сущность МО
152 8. проекция состоит из проекций, т.о. получается иерархичность агрегирования
153 
154 Внутри сущности находится иерархия по сущности-субъекту.
155 
156 Новые сущности могут создаваться в результате проецирования, т.е. по сути являются тоже проекциями.
157 
158 ------------------------------------------------------------------------------
159  Варианты взаимоотношений узлов json:
160 
161 1. Проекция в проекции
162 2. Сущность в сущности (в полях "$obj", "$rel", "$sub")
163 3. Сущность в проекции (в объекте или массиве)
164 4. Проекция в сущности (в полях "$obj", "$rel", "$sub")
165 5. Строковая или адресная ссылка на сущность в сущности (в полях "$obj", "$rel", "$sub")

168  О свойствах (атрибутах) и элементах сущности:
169 
170 У сущности кроме её внутренней троичной структуры есть свойства (её проекции) и элементы свойств.
171 Элементами сущности являются разные её свойства.
172 Элементы объекта и есть его свойства если этот объект не сущность.
173 Свойства (properties) сущности есть элементы (elements) её представления (view).
174 (json properties are elements of its view).
175 
176 https://books.google.ru/books?id=VfcX9wJEH3YC&pg=PT42&redir_esc=y&hl=ru#v=onepage&q&f=false
177 
178 Сущность это то что является субъекту при наблюдении объекта, через призму их связывающего отношения.
179 Таким образом сущьность есть результат отношения связывающего объект и субъект.
180 Субъект вмещает в себя явление сущности объекта.
181 
182 Результат исполнения сущности это проекция в триаде MVC, но т.к. МО это самокомпилирующийся иерархический многослойный MVC,
183 то субъект у сущности это родительский view вмещающий проекцию данной сущности.
184 Таким образом результат проецирования сущности сохраняется в поле "", значение которого состоит из отдельных
185 проекций агрегируемых сущностей. Следовательно если объект json имеет признак того, что это сущность,
186 то данный объект json не должен иметь в себе полей с текстовыми названиями, а только: "id", "$obj", "$rel", "$sub", "/".
187 Решением данной проблемы может быть вариант когда текстовые поля внутри объекта json будут хранить в себе кэши проекций
188 данной сущности другими контроллерами (сущностями-отношениями), например:
189 
190 {
191  "id": "this entity english name",
192  "RusView": "русское название этой сущности",
193  "jsView": "this entity java script view",
194  "C#view": "this entity C# code view",
195  "$obj": "object entity model",
196  "$rel": "relation entity model",
197  "$sub": "subject entity model"
198 }
199 
200 ------------------------------------------------------------------------------
201  Виды топологий сущностей:
202 
203 1. EntId = SubId = RelId = ObjId - статическое собственное значение,
204  сущность топологически замкнута сама на себя через себя
205 2. EntId = SubId = ObjId - динамическое собственное значение,
206  сущность топологически замкнута сама на себя,
207  через другую сущность проектор - сущность отношение
208 3. EntId = SubId - ссылка или хранилище проекции,
209  сущность топологически не замкнута сама на себя,
210  сущность сохраняет в себе проекцию другой сущности формируемой
211  другой сущность проектором - сущность отношением
212 4. EntId = ObjId - источник проекции для субъективного восприятия другими сущностями,
213  сущность топологически не замкнута сама на себя,
214  сущность формирует собственную проекцию,
215  через другую сущность проектор - сущность отношение

271  Программирование на МО
272 
273 Контекстно зависимая проекция сущности может быть получена в результате
274 исполнения (или вычисления) отношения "$rel" в контексте субъекта "$sub", объекта "$obj".
275 Доступ непосредственно к текущего значения проекции осуществляется через строковый адрес "$view".
276 Сущность может использовать контекстно зависимые местоимения в своей модели,
277 таким образом реализуется контекстная зависимость проекции сущности.
278 
279 Аналогичное выражение на языке С++ выглядело бы следующим образом:
280 
281 json context_entity_value;
282 context_entity_value = $sub->$rel( context_entity_value, $obj );
283 
284 В отличие от обычных языков программирования, в МО стек контекстов вызовов и стэк
285 локальных адресных пространств (локальные переменные) принципиально разделены.
286 
287 Локальное адресное пространство при исполнении сущности-отношения указывается всегда явно,
288 а стэк контекстов исполнения в общем случае древовидный, т.е. по сути и не является стэком,
289 что обеспечивает лёгкое многопоточное исполнение сущностей.
290 
291 Так же при исполнении сущности-отношения можно указать в каком контексте она будет исполняться, в текущем или родительском
292 аргументы тоже группируются в независимый от вызова пакет, и может быть многократно использован в контексте исполнения.
293 В обычных языках стэк контекстов (агрументы вызова, адреса возвратов из функции) объединён
294 с локальными адресными пространствами функций, поэтому очень важно четко знать где и
295 что записано в стэке, четко соблюдать call convention что бы случайно не использовать некую переменную или аргумент как адрес возврата.
