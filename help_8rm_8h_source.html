157  Самое основное отношение это отношение "#/exec", оно исполняет сущность объекта указанного

161 
163  Правила json скрипта (проекции МО в json):
164 
165  1. json представляет собой агрегат проекций сущностей т.е. иерархию МО по сущности-субъекту,
166  2. местоимение субъекта "$sub" может являеться сущностью агрегатом проекции
167  3. запуск скрипта это вычисление проекции корневой сущности
168  4. внутри корневой json-сущности лежат разные проекции её свойств, либо...
169  5. внутри корневой json проекции лежит jsonView корневой сущности
170  6. у локальных не корневых проекций субъект "$sub" равен сущности в которой они лежат
171  7. если в json объекте есть поле "$rel", то значит этот объект описывает сущность МО
172  8. проекция состоит из проекций, т.о. получается иерархичность агрегирования
173 
174  Внутри сущности находится иерархия по сущности-субъекту.
175 
176  Новые сущности могут создаваться в результате проецирования, т.е. по сути являются тоже проекциями.
177

180 
181  1. Проекция в проекции
182  2. Сущность в сущности (в полях "$obj", "$rel", "$sub")
183  3. Сущность в проекции (в объекте или массиве)
184  4. Проекция в сущности (в полях "$obj", "$rel", "$sub")
185  5. Строковая или адресная ссылка на сущность в сущности (в полях "$obj", "$rel", "$sub")

188  О свойствах (атрибутах) и элементах сущности:
189 
190  У сущности кроме её внутренней троичной структуры есть свойства (её проекции) и элементы свойств.
191  Элементами сущности являются разные её свойства.
192  Элементы объекта и есть его свойства если этот объект не сущность.
193  Свойства (properties) сущности есть элементы (elements) её представления (view).

198  Сущность это то что является субъекту при наблюдении объекта, через призму их связывающего отношения.
199  Таким образом сущьность есть результат отношения связывающего объект и субъект.
200  Субъект вмещает в себя явление сущности объекта.
201 
202  Результат исполнения сущности это проекция в триаде MVC, но т.к. МО это самокомпилирующийся иерархический многослойный MVC,
203  то субъект у сущности это родительский view вмещающий проекцию данной сущности.
204  Таким образом результат проецирования сущности сохраняется в поле "", значение которого состоит из отдельных
205  проекций агрегируемых сущностей. Следовательно если объект json имеет признак того, что это сущность,
206  то данный объект json не должен иметь в себе полей с текстовыми названиями, а только: "id", "$obj", "$rel", "$sub", "/".
207  Решением данной проблемы может быть вариант когда текстовые поля внутри объекта json будут хранить в себе кэши проекций
208  данной сущности другими контроллерами (сущностями-отношениями), например:
209 
210  {
211  "id": "this entity english name",
212  "RusView": "русское название этой сущности",
213  "jsView": "this entity java script view",
214  "C#view": "this entity C# code view",
215  "$obj": "object entity model",
216  "$rel": "relation entity model",
217  "$sub": "subject entity model"
218  }
219 
221  Виды топологий сущностей:
222 
223  1. EntId = SubId = RelId = ObjId - статическое собственное значение,
224  сущность топологически замкнута сама на себя через себя
225  2. EntId = SubId = ObjId - динамическое собственное значение,
226  сущность топологически замкнута сама на себя,
227  через другую сущность проектор - сущность отношение
228  3. EntId = SubId - ссылка или хранилище проекции,
229  сущность топологически не замкнута сама на себя,
230  сущность сохраняет в себе проекцию другой сущности формируемой
231  другой сущность проектором - сущность отношением
232  4. EntId = ObjId - источник проекции для субъективного восприятия другими сущностями,
233  сущность топологически не замкнута сама на себя,
234  сущность формирует собственную проекцию,
235  через другую сущность проектор - сущность отношение

291  Программирование на МО
292 
293  Контекстно зависимая проекция сущности может быть получена в результате
294  исполнения (или вычисления) отношения "$rel" в контексте субъекта "$sub", объекта "$obj".
295  Доступ непосредственно к текущего значения проекции осуществляется через строковый адрес "$view".
296  Сущность может использовать контекстно зависимые местоимения в своей модели,
297  таким образом реализуется контекстная зависимость проекции сущности.
298 
299  Аналогичное выражение на языке С++ выглядело бы следующим образом:
300 
301  json context_entity_value;
302  context_entity_value = $sub->$rel( context_entity_value, $obj );
303 
304  В отличие от обычных языков программирования, в МО стек контекстов вызовов и стэк
305  локальных адресных пространств (локальные переменные) принципиально разделены.
306 
307  Локальное адресное пространство при исполнении сущности-отношения указывается всегда явно,
308  а стэк контекстов исполнения в общем случае древовидный, т.е. по сути и не является стэком,
309  что обеспечивает лёгкое многопоточное исполнение сущностей.
310 
311  Так же при исполнении сущности-отношения можно указать в каком контексте она будет исполняться, в текущем или родительском
312  аргументы тоже группируются в независимый от вызова пакет, и может быть многократно использован в контексте исполнения.
313  В обычных языках стэк контекстов (агрументы вызова, адреса возвратов из функции) объединён
314  с локальными адресными пространствами функций, поэтому очень важно четко знать где и
315  что записано в стэке, четко соблюдать call convention что бы случайно не использовать некую переменную или аргумент как адрес возврата.)";
